Source Formats

Various Cantaloupe processors support several other formats than those listed here; what follows are mainly just notes.

JPEG

It should be possible to use TurboJPEG (high-level API over libjpeg-turbo with Java2dProcessor to hardware-accelerate JPEG coding. To do so, include the TurboJPEG plugin JAR in your classpath, and compile libjpeg-turbo with Java support (using the --with-java configuration option). (The author has not tried this.)

JPEG2000

JPEG2000 uses advanced compression techniques to enable fast reduced-scale and region-of-interest decoding. In theory, and with a performant decoder, JPEG2000 is well-suited for extremely large source images. Unfortunately, there are few performant decoders available for this format.

KakaduProcessor is by far the most efficient processor for this format. With a fast multi-core CPU that supports Kakadu's hardware acceleration features, it can basically cut through hundreds-of-megapixel source images like they are nothing.

GraphicsMagick can read/write JPEG2000 using JasPer, and ImageMagick using OpenJPEG. Both of these are extremely slow.

See the JaiProcessor section for notes on the mediaLib accelerator.

TIFF

GraphicsMagickProcessor and ImageMagickProcessor can both handle TIFF if the necessary delegate or plugin is installed. (See the Cantaloupe landing page.)

Pyramidal TIFF

Pyramidal TIFF is a special type of TIFF that improves TIFF's suitability for deep zooming applications. In addition to the main image, a pyramidal TIFF file contains a sequence of progressively half-sized sub-images: for example, a 10000x10000 pixel main image would include derivatives of 5000x5000 pixels, 2500x2500 pixels, 1250x1250 pixels, and so on, all embedded inside the same file. These smaller images are more efficient to read at small scales (shallow zoom levels).

Java2dProcessor and JaiProcessor both support pyramidal TIFF, which is to say that they actually do read the embedded sub-images and choose the smallest one that can fulfill the request.

All other processors that support TIFF will read pyramidal TIFFs as ordinary TIFFs, ignoring the sub-images.
